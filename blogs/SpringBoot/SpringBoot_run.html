    # 1. SpringBoot 启动过程

    0.0：new SpringApplication(primarySources) 0.1：setInitializers((Collection) getSpringFactoriesInstances(ApplicationContextInitializer.class)); 0.2：setListeners((Collection) getSpringFactoriesInstances(ApplicationListener.class));

    1.0：SpringApplicationRunListeners listeners = getRunListeners(args), Spring应用运行监听器集合，监听Spring应用启动过程中的不同阶段发布的事件 1.1：listeners.starting(); 获取支持该阶段的监听器(默认4个)发布ApplicationStartingEvent事件，默认只执行日志框架的初始化

    2.0：prepareEnvironment 环境准备 2.1：configureEnvironment 将配置(args,profiles,default)加载到运行环境中 2.2：listeners.environmentPrepared(environment) 获取支持该阶段的监听器(默认9个)执行环境准备 2.2：getApplicationListeners(event, type) 根据事件类型获取监听器并发布 ApplicationEnvironmentPreparedEvent 事件 2.2：BootstrapApplicationListener(引导)

    1. run方法进去，先 new 了一个 SpringApplication对象，这个对象的构造方法里面有两个 set 方法

      1. setInitializers (ApplicationContextInitializer.class)
        1. ApplicationContextInitializer<C extends ConfigurableApplicationContext>
        2. initialize：prepareContext方法内执行
      2. setListeners (ApplicationListener.class)
        1. ApplicationListener<E extends ApplicationEvent> extends EventListener
        2. onApplicationEvent
      3. getSpringFactoriesInstances
      4. createSpringFactoriesInstances
      5. SpringFactoriesLoader.loadFactoryNames(type, classLoader)
      6. loadSpringFactories(classLoader)
      7. 2.x版本的 SpringBoot 使用了缓存，提升了 loadSpringFactories 方法的性能

    2. ConcurrentReferenceHashMap 支持高并发，基于分段式锁实现，性能优于hashTable；和 ConcurrentHashMap设计约束一致，但是支持 key 和 value 都为 null；默认的引用类型为 软引用，适合用来做缓存，发生 gc 的时候，对象可能会被回收。

    3. 创建完SpringApplicaion对象后，执行 run 方法，先是通过 getRunListeners 来获取一个 SpringApplicaionRunListeners 对象

      1. getSpringFactoriesInstances(SpringApplicationRunListener.class)
      2. EventPublishingRunListener implements SpringApplicationRunListener

    # 2. ApplicationContextInitializer

    # 2.1 作用

    ​ ApplicationContextInitializer 接口用于在 Spring 容器刷新之前执行的一个回调函数，通常用于向 SpringBoot 容器中注入属性。

    # 2.2 内置实现类

    1. DelegatingApplicationContextInitializer
      1. 使用环境属性 context.initializer.classes 指定的初始化器(initializers)进行初始化工作，如果没有指定则什么都不做，通过它使得我们可以把自定义实现类配置在 application.properties 里成为了可能
    2. ContextIdApplicationContextInitializer
      1. 设置Spring应用上下文的ID,会参照环境属性。至于Id设置成什么值，将会参考环境属性：
        1. spring.application.name
        2. vcap.application.name
        3. spring.config.name
        4. spring.application.index
        5. vcap.application.instance_index
      2. 如果这些属性都没有，ID 使用 application。
    3. ConfigurationWarningsApplicationContextInitializer
      1. 对于一般配置错误在日志中作出警告
    4. ServerPortInfoApplicationContextInitializer
      1. 将内置 servlet容器实际使用的监听端口写入到 Environment 环境属性中。这样属性 local.server.port 就可以直接通过 @Value 注入到测试中，或者通过环境属性 Environment 获取。
    5. SharedMetadataReaderFactoryContextInitializer
      1. 创建一个 SpringBoot和ConfigurationClassPostProcessor 共用的 CachingMetadataReaderFactory对象。实现类为：ConcurrentReferenceCachingMetadataReaderFactory
    6. ConditionEvaluationReportLoggingListener
      1. 将 ConditionEvaluationReport写入日志。

    # 2.3 自定义实现类

    1. 创建三个类，实现ApplicationContextInitializer

    2. 配置方式：

      1. 在 resources/META-INF/spring.factories 中配置

      2. 在 mian 函数中添加

    # 3. ApplicationListener

    监听容器中发布的事件

    要容器中有相关事件的发布，我们就能监听到这个事件

    # 3.1 Spring 内置事件

    # 1、ContextRefreshedEvent *

    ​ ApplicationContext 被初始化或刷新时，该事件被发布。这也可以在 ConfigurableApplicationContext接口中使用 refresh() 方法中的 this.finishRefresh() 来发生。此处的初始化是指：所有的Bean被成功装载，后处理Bean被检测并激活，所有Singleton Bean 被预实例化，ApplicationContext容器已就绪可用

    # 2、ContextStartedEvent

    ​ listeners.starting()

    ​ 当使用 ConfigurableApplicationContext （ApplicationContext子接口）接口中的 start() 方法启动 ApplicationContext 时，该事件被发布。你可以调查你的数据库，或者你可以在接受到这个事件后重启任何停止的应用程序

    # 2、EmbeddedServletContainerInitializedEvent *

    ​ 当 WebServer 启动完成之后会发布该事件，来表示 WebServer 已就绪可用。

    # 3、ApplicationReadyEvent *

    ​ ApplicationContext 刷新完成后，该事件被发布。this.afterRefresh()

    # 4、ContextStoppedEvent

    ​ 当使用 ConfigurableApplicationContext 接口中的 stop() 停止 ApplicationContext 时，发布这个事件。你可以在接受到这个事件后做必要的清理的工作

    # 5、ContextClosedEvent

    ​ 当使用 ConfigurableApplicationContext 接口中的 close() 方法关闭 ApplicationContext 时，该事件被发布。一个已关闭的上下文到达生命周期末端；它不能被刷新或重启

    # 6、RequestHandledEvent

    ​ 这是一个 web-specific 事件，告诉所有 bean HTTP 请求已经被服务。只能应用于使用DispatcherServlet的Web应用。在使用Spring作为前端的MVC控制器时，当Spring处理用户请求结束后，系统会自动触发该事件

    # 3.1 原理

    1. 通过上下文环境发布一个事件：context.publishEvent(event);
    2. 由各监听器，监听具体事件，来执行具体的逻辑

    # 3.2 自定义事件发布

    @RequestMapping("event-publish")
public String eventPublish() {
  ApplicationContext applicationContext = SpringApplicionUtil.getApplicationContext();

  //发布一个事件
  String event = "我自己的事件";
  ApplicationEvent applicationEvent = new ApplicationEvent(event){};

  applicationContext.publishEvent(applicationEvent);

  return "success";
}

    # 4. EventPublishingRunListener

    • listeners.starting()：run方法进来的第一个事件 应用启动事件
    • listeners.environmentPrepared：prepareEnvironment 运行环境准备完毕事件
    • listeners.contextPrepared：prepareContext 上下文环境 准备完毕事件
    • listeners.contextLoaded：prepareContext 上下文环境 加载完毕事件
    • listeners.finished：afterRefresh方法之后，代表 SpringBoot 启动完成

    这些方法最终会遍历所有的事件监听器，依次执行 onApplicationEvent() 方法

    # 分布式事务流程

    # 分布式事务实践

    1. 提交订单（防重）
    2. 验库存/锁库存（使用分布式锁对商品ID进行加锁）
      1. 锁库存后发送延时队列定时关单（防止因为某些特殊原因导致下单事务回滚后库存未解锁）
    3. 下单事务回滚
      1. 订单逻辑错误，发送订单失败消息队列，同样对库存进行解锁
    4. 下单事务提交并支付后，减库存

    # 分布式锁原理

    1. 利用 Redis 的 SetNX 实现排他锁
    2. 利用 Redis 的过期时间，防止死锁
    3. 两个线程对同一对象加锁导致死锁，解决：可重入锁，实现：Hash(key, threadId, count) + lua 脚本
      1. 结构：(key,(threadId, count))
      2. count：计数器，实现可重入。
        1. 如果其他线程再次请求加锁，则首先判断是否为持有锁的那个线程，如果是，则直接增加count的值。
        2. 只有当计数器的值减少到0时，即所有持有锁的线程都已经释放它时，该锁才会被释放。
    4. 防止误删：A线程解锁了B线程的锁，解锁前判断是否自己的锁（线程ID）
      1. 判断和删除 使用 lua 脚本实现操作的原子性
    5. 防止业务逻辑还没有执行完毕锁就过期了：Timer定时器实现自动续期

    # RedLock和Redisson区别

    1. 实现方式不同：Redlock是基于多个Redis节点的互相协助来实现分布式锁，Redisson是基于单节点实现的
    2. 性能不同：Redlock的性能更低，因为多节点协助涉及到多个节点之间的通信
    3. 功能不同：Redisson除了实现分布式锁以外，还提供了其他的分布式功能，如：分布式集合、分布式Map

    RedLock是基于redis实现的分布式锁，它能够保证以下特性：

    互斥性：在任何时候，只能有一个客户端能够持有锁；

    避免死锁：当客户端拿到锁后，即使发生了网络分区或者客户端宕机，也不会发生死锁；（利用key的存活时间）

    容错性：只要多数节点的redis实例正常运行，就能够对外提供服务，加锁或者释放锁；

    # synchronized底层原理

    控制对类成员变量的访问。

    synchronized修饰的类或对象的所有操作都是原子的，因为在执行操作之前必须先获得类或对象的锁，直到执行完才能释放，这中间的过程无法被中断。
