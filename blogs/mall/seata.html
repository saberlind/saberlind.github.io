    # 事务

    事务的概念：逻辑上的一组操作，组成这组操作的各个逻辑单元，要么都成功，要么都失败。

    # 事务特征(ACID)

    1. Atomicity：原子性、不可分割性，一个事务中涉及的多个操作在逻辑上缺一不可。(要么都执行，要么都不执行)
    2. Consistency：一致性，事务执行之前和执行之后的数据都保持正确；即(要么都成功要么都失败)
    3. Isolation：隔离性，事务之间互不影响，事务并发执行，处理相同的数据时，多个事务不会互相干扰。
    4. Durability：持久性，保存到硬盘，数据不丢失

    # 事务隔离级别

    并发写的问题：使用mysql默认的锁机制(独占锁)

    事务并发引起读的问题：

    • 脏读： 一个事务可以读取另一个事务未提交的数据（脏数据）
    • 不可重复读 ：是指在数据库访问中，一个事务内的多次相同查询却返回了不同数据。
    • 虚读（幻读） ：指同一个事务内多次查询返回的结果集不一样，比如增加了行记录。

    备注：不可重复读对应的是修改，即update操作。幻读对应的是插入操作。幻读是不可重复读的一种特殊场景。

    解决读问题：设置事务隔离级别

    • read uncommitted(0)：以上三个问题都会出现，基本不使用。

    • read committed(2)：读已提交。解决脏读，oracle默认

    • repeatable read(4)：可重复读。解决脏读和不可重复读。mysql默认

    • Serializable(8)：解决所有问题，但是性能最低

    隔离级别越高，性能越低。

    # 相关命令

    查看全局事务隔离级别：SELECT @@global.tx_isolation

    设置全局事务隔离级别：set global transaction isolation level read committed;

    查看当前会话事务隔离级别：SELECT @@tx_isolation

    设置当前会话事务隔离级别：set session transaction isolation level read committed;

    查看mysql默认自动提交状态：select @@autocommit

    设置mysql默认自动提交状态：set autocommit = 0;【不自动提交】

    开启一个事务：start transaction; / begin;

    提交/回滚：commit / rollback

    在事务中创建一个保存点：savepoint tx1

    回滚到保存点：rollback to tx1

    # 传播行为：spring特有的

    • REQUIRED：支持当前事务，如果事务不存在，则新建事务

    • SUPPORTS：支持当前事务，如果事务不存在，就以非事务方式运行

    • MANDATORY：支持当前事务，如果事务不存在，就抛出异常

    • REQUIRES_NEW：挂起当前事务，以新事务方式运行

    • NOT_SUPPORTED：挂起当前事务，以非事务方式运行

    • NEVER：以非事务方式运行，如果事务存在，抛出异常

    • NESTED：嵌套事务，基于jdbc3.0提供的SavePoint技术实现

    常用的：

    REQUIRED：一个事务，要么成功，要么失败。

    REQUIRES_NEW：两个不同的事务，彼此之间没有关系。一个事务失败了不影响另一个事务。

    @Transactional(propagation = Propagation.REQUIRED)

    # 回滚策略

    所有的受检异常（编译时异常），都不回滚

    所有的不受检异常（运行时异常），都回滚

    自定义回滚策略：

    rollbackFor rollbackForClassName noRollbackFor noRollbackForClassName

    # 分布式事务

    # 理论

    # CAP

    分布式存储系统的CAP原理(分布式系统的三个指标)

    1. Consistency(一致性)：在分布式系统中的所有数据备份，在同一时刻是否同样的值。
      1. 强一致性：写完之后，立马可以读取到最新的值
      2. 弱一致性：中间状态
      3. 最终一致性：弱一致性的特殊情况
    2. Availability(可用性)：在集群中一部分节点故障后，集群整体是否还能响应客户端的读写请求。
    3. Partition tolerance(分区容错性)：大多数分布式系统都发布在多个子网络。每个子网络叫做一个区(partition)，区间通信可能失败

    CAP理论就是说在分布式存储系统中，最多只能实现其中两点，由于当前的网络硬件肯定会出现延迟丢包等问题，所以分区容错性是无法避免的。要么AP（mysql redis）、要么 CP（zookeeper）。

    # BASE(基于AP)

    BASE模型是传统ACID模型的反面，不同于ACID，BASE强调牺牲高一致性，从而获得可用性。

    BA：Basically Available，基本可用，保证核心可用

    S：Soft Status，软状态。允许中间状态存在

    E：Eventually Consistency，最终一致。

    # 解决方案

    # 1.2PC

    2：两个阶段

    P：Prepare预提交阶段

    C：Commit提交阶段

    基于2PC的XA协议，目前主流的数据库都支持了，免费开源(mysql)的支持不够好。

    优点：实现简单

    缺点：

    1. mysql的支持不够完善
    2. 由于两个阶段需要多次网络传输，通信时间太长，事务时间相对变长，导致锁定资源的时间变长了，最终性能不高。

    # 2.TCC

    T：Try，预检资源并锁定资源

    C：Confirm，执行业务操作

    C：Cancel，取消或者回滚事务

    补偿性事务解决方案，编程式分布式事务解决方案

    缺点：

    1. 代码量非常庞大
    2. 在业务逻辑复杂的情况下，很难把握补偿点
    3. 幂等性很难保证

    # 3.MQ最终一致性

    可靠性较差，性能较高

    # 总结

    适用于高并发：最终一致

    低并发基本一致：二阶段提交

    高并发强一致：暂无解决方案

    # Seata

    AT模式基于2PC提交

    TCC模式

    订单：MQ最终一致性 + 补偿机制

    全局事务：一个分布式事务就是一个全局事务，由一系列的分支事务组成

    分支事务：一个分子事务就是一个本地事务

    # 三个基本组件

    TC：Transaction Coordinator，事务协调器，监控全局事务和分支事务的执行状态，驱动全局事务的提交或回滚。

    TM：Transaction Manager，事务管理器，定义一个全局事务的范围，开启一个全局事务，并最终发起全局事务的提交或回滚

    RM：Resource Manager，资源管理器，监控分支事务的工作资源，向TC注册分支事务并汇报分支事务的执行状态，驱动分支事务的提交或回滚。

    # 工作流程

    1. TM向TC注册一个全局事务，TC生成一个全局事务，并返回一个全局事务的唯一标识XID
    2. XID会在微服务的调用链路中传播
    3. RM向TC注册一个本地事务，作为XID相应的全局事务中的一个分支事务
    4. TM向TC发起XID响应的全局事务的提交或回滚
    5. TC驱动RM完成XID相应的全局事务下的所有分支事务的提交和回滚

    TM向TC发起全局事务的提交或回滚，TC驱动RM对分支事务进行提交和回滚。最终由RM完成分支事务的提交和回滚。

    # DataSourceConfig.java

    每一个微服务原来自己的数据源都必须使用DataSourceProxy代理，这样seata才能掌控所有事务。

    @Configuration
public class DataSourceConfig {

    /**
     * 需要将 DataSourceProxy 设置为主数据源，否则事务无法回滚
     */
    @Bean("dataSource")
    public DataSource dataSource(
            @Value("${spring.datasource.driver-class-name}")String driverClassName,
            @Value("${spring.datasource.url}")String url,
            @Value("${spring.datasource.username}")String username,
            @Value("${spring.datasource.password}")String password
    ) {
        HikariDataSource hikariDataSource = new HikariDataSource();
        hikariDataSource.setJdbcUrl(url);
        hikariDataSource.setDriverClassName(driverClassName);
        hikariDataSource.setUsername(username);
        hikariDataSource.setPassword(password);
        return new DataSourceProxy(hikariDataSource);
    }
}
